# VitalAccess — Development Tasks

## 1. Project Scaffolding & Configuration

1.1. Create a new Flutter project in the current directory (or reinitialize if already created) targeting Android and iOS.

1.2. Add all required dependencies to `pubspec.yaml`:
   - `camera` — front camera access
   - `dio` — HTTP client for Claude API calls
   - `share_plus` — system share sheet
   - `flutter_dotenv` — environment variable management for API keys
   - `uuid` — session ID generation
   - `provider` or `riverpod` — state management (pick one, keep it simple)
   - `google_fonts` — typography
   - `pdf` — PDF generation for health summary
   - `path_provider` — file system access for saving PDFs

1.3. Create the folder structure:
   ```
   lib/
   ├── main.dart
   ├── app.dart
   ├── config/
   │   ├── theme.dart
   │   └── constants.dart
   ├── models/
   │   ├── vital_scan_result.dart
   │   ├── symptom_intake.dart
   │   ├── triage_result.dart
   │   ├── health_summary.dart
   │   └── chat_message.dart
   ├── services/
   │   ├── claude_service.dart
   │   ├── vitals_service.dart
   │   └── share_service.dart
   ├── screens/
   │   ├── language_select_screen.dart
   │   ├── scan_screen.dart
   │   ├── symptom_chat_screen.dart
   │   ├── processing_screen.dart
   │   └── result_screen.dart
   └── widgets/
       ├── vitals_display.dart
       ├── chat_bubble.dart
       ├── urgency_badge.dart
       └── health_summary_card.dart
   ```

1.4. Create a `.env` file (and add it to `.gitignore`) with a placeholder for `ANTHROPIC_API_KEY`.

1.5. Set up the app theme in `config/theme.dart`:
   - Define a clean, medical-feeling color palette (white/light background, teal/blue primary accent)
   - Define the four urgency colors: red (emergency), amber/yellow (urgent), green (routine), blue (self-care)
   - Set up text themes with clear hierarchy (large vitals numbers, readable body text)

1.6. Set up routing/navigation in `app.dart`:
   - Define named routes for all 5 screens: `/language`, `/scan`, `/chat`, `/processing`, `/result`
   - Set `/language` as the initial route
   - Use `MaterialApp` with the custom theme

1.7. Set up a top-level state management solution (e.g., a `SessionProvider` or `SessionNotifier`) that holds:
   - Selected language (`String`)
   - `VitalScanResult?`
   - `SymptomIntake?`
   - `TriageResult?`
   - `HealthSummary?`
   - A method to reset the session for starting over

## 2. Data Models

2.1. Create `models/chat_message.dart`:
   - Fields: `String role` (user/assistant), `String content`, `DateTime timestamp`
   - Add `toJson()` and `fromJson()` methods

2.2. Create `models/vital_scan_result.dart`:
   - Fields: `double heartRate`, `double heartRateVariability`, `double respiratoryRate`, `DateTime timestamp`
   - Add `toJson()` method for passing to Claude API
   - Add a `toReadableMap()` method that returns a human-friendly map like `{"Heart Rate": "88 bpm", ...}`

2.3. Create `models/symptom_intake.dart`:
   - Fields: `List<ChatMessage> conversation`, `String structuredSummary`
   - `structuredSummary` is populated after the chat ends, generated by Claude summarizing the conversation

2.4. Create `models/triage_result.dart`:
   - Define `enum UrgencyLevel { emergency, urgent, routine, selfCare }` with extension methods for:
     - `color` → returns the appropriate `Color`
     - `label` → returns the display string (e.g., "Seek emergency care now")
     - `icon` → returns an appropriate `IconData`
   - Fields: `UrgencyLevel urgency`, `String clinicalReasoning`, `String plainExplanation`, `String watchFor`
   - Add a `fromJson(Map<String, dynamic>)` factory that parses the Claude API JSON response, matching urgency strings to the enum

2.5. Create `models/health_summary.dart`:
   - Fields: `String sessionId`, `DateTime timestamp`, `String language`, `VitalScanResult vitals`, `SymptomIntake symptoms`, `TriageResult triage`
   - `sessionId` generated via `uuid` package on creation
   - Add a `toShareableText()` method that formats the entire summary as a readable plain-text string (for sharing via WhatsApp/SMS)

## 5. Claude API Service

5.1. Create `services/claude_service.dart` with a class `ClaudeService`:
   - Constructor takes the API key from environment
   - Use `dio` for HTTP requests to `https://api.anthropic.com/v1/messages`
   - Set required headers: `x-api-key`, `anthropic-version: 2023-06-01`, `content-type: application/json`
   - Use model `claude-sonnet-4-20250514`

5.2. Implement the symptom intake chat method:
   - Method: `Future<String> sendSymptomMessage(List<ChatMessage> conversationHistory, String userMessage, String language, VitalScanResult vitals)`
   - System prompt must:
     - Define Claude as a structured symptom collector for a health triage app
     - Instruct it to ask one clear question at a time, branching based on answers
     - Tell it to respond in the user's selected language
     - Provide the vitals context so it can ask relevant follow-up questions
     - Explicitly state: "You are collecting symptoms for triage. Never diagnose. Never name diseases. Ask 5-8 questions maximum."
     - Instruct it to start with "Do you have any symptoms or concerns right now?" as the first message
     - After sufficient questions, instruct it to end with a special token like `[INTAKE_COMPLETE]` followed by a structured JSON summary of all collected symptoms
   - Send the full conversation history each time (Claude Messages API is stateless)
   - Parse the response, detect `[INTAKE_COMPLETE]` to know when intake is done

5.3. Implement the triage pipeline method:
   - Method: `Future<TriageResult> runTriagePipeline(VitalScanResult vitals, String symptomSummary, String language, {int? age, String? sex})`
   - Make a single Claude API call with a system prompt that instructs Claude to perform all 4 reasoning steps in sequence:
     - Step 1 — Vitals Interpretation: Analyze HR, HRV, RR against clinical baselines for the given age/sex
     - Step 2 — Symptom Assessment: Map symptom clusters to clinical patterns
     - Step 3 — Triage Classification: Fuse steps 1+2 into one urgency level (emergency/urgent/routine/selfCare)
     - Step 4 — Plain-Language Explanation: Rewrite everything at a 6th-grade reading level in the user's language
   - System prompt must explicitly state: "You are a triage classifier. Output urgency levels only. Never name specific diseases or conditions. Never say 'you have X.' Always say 'this suggests you should see a doctor because...'"
   - Require the response in a strict JSON format:
     ```json
     {
       "urgency": "emergency|urgent|routine|selfCare",
       "clinicalReasoning": "...",
       "plainExplanation": "...",
       "watchFor": "..."
     }
     ```
   - Parse the JSON response into a `TriageResult` object
   - Add error handling: if parsing fails, default to `urgent` with a generic "please see a healthcare provider" message

## 10. Error Handling & Edge Cases

10.1. Handle camera permission denial on `ScanScreen`:
   - Show a friendly message explaining why camera access is needed
   - Provide a button to open app settings
   - Provide a "Continue without scan" option that uses mock vitals (for demo resilience)

10.2. Handle Claude API failures in `ClaudeService`:
   - Network errors: show a retry-able error with "Check your connection" message
   - API key errors (401): show "API key not configured" message
   - Rate limiting (429): show "Please wait a moment and try again"
   - Timeout: set a 30-second timeout on all API calls, show retry on timeout
   - Malformed response: log the raw response, fall back to safe defaults

10.3. Handle the case where the symptom chat stalls (Claude doesn't produce `[INTAKE_COMPLETE]` after 10 messages):
   - After 10 exchanges, show a "Finish intake" button that forces a summary request to Claude
   - Send a special message to Claude: "Please summarize the symptoms collected so far and end with [INTAKE_COMPLETE]"
